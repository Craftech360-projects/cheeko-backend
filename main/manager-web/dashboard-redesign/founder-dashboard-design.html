What I found in the current dashboard

Your instinct is right — it's an ops console, not a founder view. Four specific things explain the cluttered feel:

Fragmented
Analytics lives in 4 different pages

Home KPI strip, Game Analytics, Token Analytics and Active Devices each tell a partial story. There is no single "how is Cheeko doing?" screen.

Nav at capacity
10 top-bar tabs + a 6-item dropdown

The flat pill nav already scrolls horizontally. Founder signal (engagement, cost) sits beside plumbing (params, dictionaries, OTA).

No chart engine
Charts are hand-rolled CSS bars

No ECharts/Chart.js installed — so today's pages lean on stat cards and giant tables. Real charts unlock trends, not just totals.

One honest gap
No like/dislike signal is captured

Nothing in the schema records explicit reactions. But strong proxies exist: repeat plays, card-tap counts, session length, completion vs. abandon.

A

Morning Pulse

One warm, glanceable overview that becomes your landing page. Five numbers up top, then the three founder questions in order: where is time spent → what do kids love → what needs attention. Everything clicks through to the existing deep pages.

Why this works

  • Answers the founder loop in one screen: healthy? → where's the time going? → what's loved/ignored? → what needs action?
  • Every widget maps to data you already collect — usage rollups, tap logs, game plays, session summaries, token costs.
  • Nav shrinks from 16 destinations to 6 groups; all current admin pages survive under "Operate".
  • Trends over totals: every number carries a vs-last-week delta, so "is this good?" answers itself.

Trade-offs

  • It's a daily-driver dashboard, not a wall display — no live ticking (that's Option B's trick).
  • Needs a handful of small fleet-level aggregation endpoints (detailed in the data map below).
  • Topic chips need one new backend job to mine session summaries — v1 can ship with top content/games only.

Why this works

  • Feels alive: the activity feed turns telemetry into a story you can watch — great founder dopamine and brilliant for demos/investors.
  • Ops truth on one wall: online now, battery, latency, spend-vs-budget — the "is anything on fire?" answer.
  • The hourly heat strip finds your true play windows, which should drive content-drop timing and push notifications.

Trade-offs

  • Most build effort: needs polling or a websocket layer plus careful IST bucketing; the feed needs an event union endpoint.
  • Live views reward watching, not deciding — weekly trends and content insight get less room.
  • Transcript snippets on a shared screen need a privacy stance (first names only, blur mode for demos).
C

Not a wall of charts — a two-minute morning read, auto-written from the same aggregates. The dashboard tells you what changed and why it matters; deep pages are one click away. Doubles as the daily email you already built plumbing for.

Why this works

  • Zero chart-reading effort: the system does the analysis and writes the "so what" — closest to a chief-of-staff.
  • Cheapest to ship — it's a rendering of the same aggregates as Option A, and your email-report cron + config UI already exist.
  • Reads beautifully on a phone over morning coffee; charts optional.

Trade-offs

  • Not exploratory — when the brief raises a question ("why did radio drop?"), you still need Option A's pages to dig.
  • Insight-picking logic needs tuning time to avoid crying wolf or going bland.
  • Some headlines (topic mining, quote surfacing) want a small LLM summarization job — token cost is tiny but real.

The Brief · mobile

How it's built

  • Responsive, not a separate app: the new pages use fluid grids — sidebar becomes a bottom tab bar under 768px, KPI rows go 2×2, tables render as cards.
  • Add-to-home-screen: the web build already carries workbox/PWA plumbing — we turn it on with a manifest + icons, so it opens full-screen like an app.
  • Same data, same auth — nothing new on the backend for mobile.

Honest limits

  • The existing admin pages (RFID studio, OTA, params) stay desktop-first — they're work tools. Mobile focus is Overview, Families, Brief, and alerts.
  • Mission Control (Option B) is a TV/desktop experience; on a phone it degrades to the Overview's live strip.

Recommendation: ship A as the new home, grow C out of it, borrow B's best row.

Phase 1 — Morning Pulse (Option A) becomes the post-login landing page, with the nav regrouped to Overview · Engagement · Content & Games · Conversations · Families · Costs · Operate — and it ships with family search + the Family 360 profile and mobile-responsive layouts from day one. All existing admin pages stay, one level down. This alone fixes "cluttered" and gives you the glance you asked for.

Phase 2 — The Brief (Option C) reuses the exact same aggregates as a web page + your existing daily-email pipeline. Phase 3 folds Mission Control's best ideas — the "online now" strip and live activity feed — into the top of the Overview, instead of maintaining a separate wall app.

You get all three experiences over time while building each aggregate exactly once.

Data reality check — what powers each widget

Grounded in manager-api-node as it exists on this branch. Green is wired today; blue is a small new endpoint over data you already store; orange needs new instrumentation.

Ready today

Active toys, sessions, accuracy, TTFT, cost

/analytics/today|month/device-count, /analytics/dashboard/summary, /sessions-per-day, /game-accuracy, /ttft-trend, /usage/analytics/totals — plus the per-device drill-down (RFID / chat / games / radio) already behind Active Devices.

Ready today

New families & fleet growth

/admin/stats/overview, /admin/stats/users, /admin/stats/devices already return registrations per day — currently rendered nowhere.

Small new endpoint

"Where kids spend time" by feature

Data exists per device in device_usage_daily (AI talk / cards / games / radio seconds). Needs one fleet-level daily-sum endpoint, ~30 lines of Prisma.

Small new endpoint

Card-pack & game leaderboards

rfid_card_tap_log has pack names per tap; device_games_played has per-play scores/duration. Group-by endpoints with week-over-week deltas. (Join taps on MAC, dates in IST — your team's plan doc already nails these pitfalls.)

Small new endpoint

Quiet toys / churn risk

ai_device.last_connected_at + usage rollups → "active last month, silent 7+ days" list with parent display names.

New job + endpoint

"What kids talk about"

voice_session_summaries already stores an LLM summary per session. A nightly job extracts topics/quotes across the fleet; endpoint serves the top list. Optional tiny LLM cost, or keyword-only for free.

Roadmap — not captured yet

Explicit likes / dislikes

No reaction signal exists anywhere in the schema. V1 uses honest proxies (repeat plays, taps, completion, session length). If you want the real thing: log a "played again / stopped early / kid said more-more" event from the agent — one table, one firmware/agent hook.

Roadmap

Sentiment on conversations

Transcripts exist (voice_session_messages), but no emotion/sentiment scoring is stored. Could ride along with the nightly topics job later.

Small new endpoint

Family search

The pieces exist — /admin/users, /admin/device/all?search, kid profiles — and the header already has a search component. One unified /admin/search?q= across parent / kid / MAC / alias makes it feel instant.

Small new endpoint

Family 360 profile

Per-device analytics & progress endpoints (/admin/device/:mac/analytics/*, /progress/*), chat summaries, and quota all exist. One aggregator returns the whole profile in a single call.

Frontend build

Mobile & add-to-home-screen

Responsive layouts for the new pages (bottom tab bar under 768px) + enabling the PWA plumbing (workbox is already in the build) with a manifest and icons. No backend work.
